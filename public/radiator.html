DLite Information Page

DLite is the Motor Votor effort to bring the DL 44 application process online. The project is a joint effort between the DMV, Secretary of State and the California Department of Technology. Below are some links to important tools and documents that we are using to get our work done.

  • Engineering

    • Staging site

      The staging site automatically deploys. First, developers merge a group of commits into the master branch of our source code. Our continuous integration tests kick off, and if the tests are successful the code is deployed directly to our staging server on heroku.

      That means that this site is the bleeding edge of development, and features are being released as they are finished.

    • Stories

      Product writes detailed stories for the development team. Features, bugs and chores are built in the icebox. When they are prioritized and ordered they are put into the current iteration. Developers choose full-stack stories from the top of the queue and pair or work on them is solo, depending on the complexity of the task. Points on the stories are estimates of complexity. Velocity is a measure of how many points can be done in a period of time. The tool, Pivotal Tracker, estimates based on recent velocity what will get done in the coming sprint.

    • Source code

      Our source code is stored on GitHub, an online source code repository, and it is part of the state's enterprise account. The source code is open source, meaning it can be copied and reused without permission. It is also public, meaning anyone can see the code and comment on it.

    • Continuous integration tests

      This project uses a continuous integration tool Travis.org that is free for open source projects. This testing tool monitors our GitHub repository for changes and kicks off the test suite. It reports results back to GitHub so that developers have centralized access when reviewing code.

      Code coverage is also being automatically broadcast from the test suite. The tools for this integration are still being tuned so that we have the best information about coverage on the acceptance, server and client levels.

    • Code metrics

      Engineering quality is monitored by doing static analysis. This checks for linting errors, complexity, and duplication. This combined with code coverage gives a good measure of the overall code quality. The tool rates the source code on a 0-4 scale, like a grade point average. The tool also uses a green to red color scale.
